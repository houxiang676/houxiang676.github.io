数据库

数据库基本概念

信息= 数据+数据处理

计算机数据管理历经三个阶段： 人工管理阶段 、文件系统阶段、数据库系统阶段

数据管理者 数据面向的对象 数据的共享程度 数据的独立性 最小数据单位 结构化
人工管理阶段 人 某一应用程序 不共享，冗余度极大 不独立，完全依赖程序 - 无结构
文件系统阶段 文件系统 某一应用程序 共享差，冗余度大 独立性差 记录 记录内有结构
整机无结构
数据库系统阶段 数据库管理系统 整个应用系统 共享性高，冗余度小 具有高度的物理独立性
和逻辑独立性		 记录中的数据项 整体结构化
用数据模型描述

数据库基础依赖文件系统作为最底层的存储，也可以会实现一些文件系统的功能，文件系统也可能会用到一些简单的数据库功能。

数据模型，模型是对现实世界的模拟和抽象，在数据库技术中，表示实体类型以及实体类型间联系的模型0称为为数据模型。

数据模型满足三方面：

  • 比较真实的模型现实世界
  • 容易为人所理解
  • 便于在计算机上实现

数据处理的三个世界： 现实世界–>信息世界（概念模型）–>计算机世界（DBMS支持的数据模型）

数据要素：数据结构、数据操作、数据的完整性约束条件

模型数据的分类

  1. 概念数据模型

    E-R图，码：唯一标识实体的属性集称为码

  2. 结构数据模型

    数据项

    记录

    文件

    1. 层次模型

      用树结构来表示数据之间的关系，除根节点外，其他节点有且仅有一个父节点

    2. 网状模型

      用图结构表示数据之间的关系，特点是，允许节点有多余一个的父节点，可以有一个以上的节点没有父节点

    3. 关系模型

      用二维表格结构来表示实体以及实体之间联系的数据模型。

      • 描述一致性
      • 可直接表示多对多的联系
      • 关系必须是规范化的关系，即每个属性是不可分的数据项，不允许表中有表
      • 关系模型师建立在数学概念基础上的，有较强的理论根据
      • 关系模型的存取路径对用户匿名，没从而具有更高的数据独立性、更好的安全保密性

      SQL，Oracle

    4. 面向对象模型

      感觉有点像多模态，因为传统数据模型在表示图形、图像。声音等多媒体数据以及空间数据，时态数据和超文本数据这类复杂数据是，已明显表示出其建模能力的不足。面向对象是用类层次结构来宝石数据之间的联系

建立求解问题的数据模型分为三类：

  • 用户使用的数据视图叫做外模型，局部的逻辑数据视图，表示用户所理解的实体，实体属性和实体关系
  • 全局的逻辑数据实图叫做概念模型，是数据库管理人员所看到的实体、实体属性和实习之间的联系
  • 模型的物理存储模型叫做内模型

相应的，整个数据库系统分为三层 外层、概念层和内层

  • 外模式（子模式和用户模式）：之定义外模型的模式，他是数据库用户能够看见的使用的局部数据的逻辑结构和特征的描述，是数据库用户的数据视图，是与某一应用程序有关的数据的逻辑表示，外模式用DBMS的外模式描述语言DDL来定义
  • 模式（概念、逻辑模式）：定义概念模型的的模式
  • 内模式（物理模式、存储模式）：定义内模型的模型，是数据物理机构和存储方式的描述。

DDL数据定义语言

修改表结构的语句： ALTER TABLE

数据分组报表：第一个分组是部门，性别 基本工资的索引表达式：部门+性别+STR

定位第一条记录上的命令是 GO TOP

关系模型中，实现“关系中不许与出现相同的元祖”的约束是通过 主键

下面是不同类型的数据模型及其特点和示例的表格：

数据模型 特点 优点 缺点 示例
层次模型 数据以树形结构存储，父子节点之间存在严格的层次关系 数据结构简单，适合层次性强的数据 父子关系严格，无法处理多对多关系 早期的 IBM IMS
网状模型 数据以图的形式存储，支持复杂的多对多连接 能处理复杂的多对多关系，灵活性较高 结构复杂，维护难度较大 CODASYL 数据库
关系模型 数据以二维表形式存储，支持通过主键和外键建立表与表之间的关系 结构简单，支持 SQL 查询语言，应用广泛 对大规模数据操作效率较低 MySQL、Oracle、PostgreSQL
面向对象模型 数据通过对象表示，支持类和继承等面向对象编程特性 能处理复杂数据结构，适合对象复杂的应用场景 实现复杂，查询语言不如 SQL 成熟 GemStone、db4o
文档模型 数据以文档形式存储，支持半结构化和非结构化数据 灵活性高，文档之间无严格的结构要求 缺乏关系模型的完整性约束和事务处理能力 MongoDB、CouchDB
键值对模型 以键值对形式存储数据，键唯一对应一个值 简单高效，适用于分布式系统和缓存场景 不适合处理复杂的查询和关系数据 Redis、DynamoDB
列存储模型 数据按列存储，支持列式操作和大规模数据处理 适合大规模数据分析和快速读写操作 在复杂查询中不如关系模型灵活 HBase、Cassandra
图模型 数据以图的形式存储，节点表示实体，边表示实体之间的关系 适合处理高度互联的数据，如社交网络和推荐系统 对海量节点和边的查询效率较低 Neo4j、JanusGraph
时序模型 专门用于处理时间序列数据，每个数据点与时间戳相关 优化了时间序列数据的存储和检索，适合高频数据的处理和分析 功能专一，适用范围较窄 InfluxDB、TimescaleDB

这些数据模型适合不同的应用场景，选择合适的数据模型可以提高系统的效率和灵活性。
